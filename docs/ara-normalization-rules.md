# ARA Normalization Rules

Use these rules when importing a blueprint generated by another model or tool.

## Goals

- preserve user intent
- rebuild structure to the local schema
- record ambiguity instead of hiding it
- avoid silent authority rewrites

## Required steps

1. Detect source type and source provenance.
2. Extract authority, workflow, stages, modules, and verification intent.
3. Rebuild the package into the local schema.
4. Produce a normalization report that records:
   - source files
   - inferred fields
   - dropped fields
   - semantic hints
   - semantic risks
   - unresolved ambiguities
5. Recompute module language assignments against local policy.
6. Recompile the contract and readiness bundle.

If no concrete source file exists, the report should still record a stable synthetic source such as a source summary marker. Authoring must remain auditable even for generated-first flows.

The runtime now enforces the embedded normalization policy:

- only supported import source formats may enter normalization
- supported structured `json` and `toml` sources are parsed into local blueprint sections before normalization continues
- nested structured containers such as `requirements`, `governance`, `workflow`, and stage-local `verification` blocks are traversed recursively instead of only reading top-level keys
- markdown imports now extract semantic intent from inline labeled prose such as `Acceptance Criteria: ...` and `Constraints: ...`, not only from explicit section headings
- markdown imports also retain multiline inline semantic blocks, so labels like `Acceptance Criteria:` or `Validation Plan:` can absorb following bullet items and continuation lines instead of losing them as plain prose
- richer external semantics such as acceptance criteria, constraints, assumptions, out-of-scope blocks, and phase-style collections are mapped into local schema sections when possible
- structured `json` and `toml` `stages` arrays are normalized into explicit stage order and stage-specific stage doc detail
- phase, milestone, and step-style structured collections may also be normalized into stage order and stage detail when they carry deterministic stage ids
- markdown phase and milestone lists may also be normalized into stage order and stage-scoped detail when each stage item carries a deterministic stage id and labeled child detail such as `Deliverables:` or `Validation Plan:`
- nested markdown stage headings under phase-style sections may also be normalized into stage order and stage-scoped detail when subheadings such as `Deliverables` or `Validation Plan` appear beneath deterministic stage headings
- cross-source markdown blocks may also contribute stage-scoped detail when one block establishes deterministic stage ids and a later block uses stage alias headings such as `## Foundation` or `## Hardening`
- file stems such as `foundation-notes.md`, `hardening-validation-plan.md`, or similar stage-alias variants may also establish stage scope for recognized headings and inline labels inside that source file
- document-level stage metadata such as front matter `stage: foundation`, list metadata like `- Milestone: foundation`, or inline metadata like `Stage ID: stage-02` and `phase_id = "stage-02"` may also establish stage scope for recognized headings and inline labels inside that source file
- normalization now emits source-first semantic frames and semantic clusters so the runtime can preserve both normalized values, semantic merge patterns, and where those semantics came from
- semantic confidence is now tiered, so exact mappings, aliases, inline heuristics, cross-source evidence, and path-alias evidence do not collapse into one generic confidence bucket
- stage-scoped indirect evidence is now checked for semantic divergence by normalized source, so conflicting deliverables or verification intent from different source files remain visible as risks and unresolved ambiguities
- authority rewrites are blocked when policy disables them
- implicit stage creation is rejected for import and update flows when policy requires deterministic stage ids
- unresolved ambiguities must remain recorded in the normalization report instead of being silently dropped

## Update and recompile rules

- `update-blueprint` should re-read the existing workspace blueprint bundle before applying update input.
- Stage order updates should merge without dropping previously known stages.
- When a workspace bundle has stage documents but no reliable workflow stage order, stage metadata may be used to recover the stage graph.
- Stage source-file blocks under `blueprint/stages/` should preserve stage-specific deliverables, verification, review focus, and repair routing when the source provides them.
- List-like sections such as deliverables may be union-merged when both sources agree on intent.
- update flows should emit patch-level operations so the merge strategy is machine-readable instead of being visible only in the final merged result.
- patch-level update intent should also be emitted as a dedicated `patch-plan.json` artifact so workspace validation can compare patch truth against `change-report.json` and fall back for older workspaces that only carry `change-report.json`.
- update patch plans should be executable against the pre-update workspace blueprint source; if replay cannot reproduce the merged result, normalization must stop instead of trusting the merge.
- Authority sections such as `Purpose` must not be silently rewritten. Those updates must stop with `ARA-3000`.
- `recompile-contract` should prefer workspace blueprint docs as the source of truth and regenerate the machine contract from them.
- Stage document file names should stay deterministic, but path collisions must be disambiguated instead of collapsing multiple stages into one file.

## Hard stops

- source blueprint would require changing authority truth
- stage boundaries are irreconcilable
- module responsibilities conflict with local language policy
- source material is too incomplete to normalize deterministically
